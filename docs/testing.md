# OpenEn — 测试与验收文档

## 测试理念

**真正的验收标准是用户体验**：打开推特能不能拆、拆得对不对、快不快。单元测试保证核心逻辑不出低级 bug，浏览器验收测试保证东西真的能跑。两者缺一不可。

开发节奏：**做一步、测一步**。不攒到最后再测。

## 测试分类

| 类型 | 工具 | 谁跑 | 验什么 |
|------|------|------|--------|
| 单元测试 | Vitest | Claude 自动 | 纯逻辑：分块规则、复杂度评分、请求格式、数据层 |
| 浏览器验收 | Chrome CDP 脚本 | Claude 自动 | 真实效果：扩展加载后页面行为、DOM 变化、性能 |
| 人工确认 | 用户操作 | 用户 | 视觉效果、交互体感、主观质量判断 |

## 测试基础设施

### 单元测试（Vitest）

```bash
npm test              # 跑全部
npm test -- scan-rules # 跑指定文件
```

测试文件放在 `src/__tests__/` 下，文件名对应被测模块（如 `scan-rules.test.ts`）。

### 浏览器验收测试（Chrome CDP）

**调试用 Chrome 配置**

独立配置目录 `~/.chrome-debug-profile/`，X/Twitter 已登录，与用户主力 Chrome 互不影响。

启动命令：
```bash
/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome \
  --remote-debugging-port=9222 \
  --user-data-dir="$HOME/.chrome-debug-profile" \
  &>/dev/null &
```

验证连接：`curl -s http://localhost:9222/json/version`

> Chrome 新版本要求 `--remote-debugging-port` 必须搭配非默认 `--user-data-dir`，否则端口不生效。

**加载扩展进行测试**

构建后用以下命令启动（同时加载扩展）：
```bash
/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome \
  --remote-debugging-port=9222 \
  --user-data-dir="$HOME/.chrome-debug-profile" \
  --load-extension=./dist \
  &>/dev/null &
```

**验收脚本**

验收脚本放在 `tests/acceptance/` 下，用 Python + websockets 通过 CDP 连接浏览器，执行 JS 检查 DOM 和性能。每个脚本可独立运行，输出 PASS/FAIL。

```bash
python3 tests/acceptance/scan-mode.py   # 扫读模式验收
python3 tests/acceptance/deep-read.py   # 细读模式验收
```

## 各开发步骤的验收标准

### 第 1 步：项目骨架

| 验收项 | 类型 | 通过标准 |
|--------|------|----------|
| 构建成功 | 自动 | `npm run build` 零错误，dist/ 下生成 content.js、background.js、popup.js、options.js |
| 扩展可加载 | 浏览器验收 | Chrome 加载 dist/ 为解压扩展，chrome://extensions 无报错 |

### 第 2 步：复制可复用代码

| 验收项 | 类型 | 通过标准 |
|--------|------|----------|
| 类型检查 | 自动 | `tsc --noEmit` 零错误 |
| 构建成功 | 自动 | `npm run build` 零错误（复制的代码不破坏构建） |

### 第 3 步：LLM 适配层

| 验收项 | 类型 | 通过标准 |
|--------|------|----------|
| Gemini 请求格式 | 单元测试 | 构造的请求体符合 Gemini API schema |
| OpenAI 请求格式 | 单元测试 | 构造的请求体符合 OpenAI chat completions schema |
| 响应解析-正常 | 单元测试 | 正确提取分块结果和生词释义 |
| 响应解析-异常 | 单元测试 | 畸形 JSON、空响应、超时均有明确错误，不崩溃 |
| 端到端调用 | 浏览器验收 | Service Worker 中发送一个硬编码句子，console 打印出 LLM 返回的分块结果 |

### 第 4 步：细读模式

| 验收项 | 类型 | 通过标准 |
|--------|------|----------|
| 复杂度评分 | 单元测试 | 10 个测试句子的评分符合预期（简单句 < 阈值，复杂句 ≥ 阈值） |
| 长度权重生效 | 单元测试 | 同样语法复杂度的句子，40 词比 15 词得分明显更高 |
| 页面分块生效 | 浏览器验收 | 打开英文文章页 → 复杂句被自动拆分渲染 → DOM 中出现分块标记元素 |
| 简单句不过度拆 | 浏览器验收 | 同一页面中，简单短句保持原样，未被拆分 |
| 手动触发按钮 | 浏览器验收 | 未自动拆分的长句 hover 时出现触发按钮 |
| 视觉效果 | 人工确认 | 缩进层级清晰，阅读体验自然 |

### 第 5 步：扫读模式（核心体验）

> 这是最重要的验收节点。用户的核心体验——"打开推特，句子有没有被拆"——在这一步验证。

| 验收项 | 类型 | 通过标准 |
|--------|------|----------|
| 本地拆分-并列句 | 单元测试 | "A and B but C" 类句子在 and/but 处断行 |
| 本地拆分-转折 | 单元测试 | however/although/while 等处正确断行 |
| 本地拆分-条件 | 单元测试 | if/unless/when 等处正确断行 |
| 本地拆分-从句 | 单元测试 | which/who/that 引导的从句正确断行 |
| 本地拆分-短句不拆 | 单元测试 | 低于长度阈值的句子原样返回 |
| 本地拆分-缩进层级 | 单元测试 | 输出保留主句/从句的层级关系 |
| 复杂句降级 | 单元测试 | 多层嵌套句被标记为"需要 LLM"，不由本地处理 |
| 模式自动判断 | 单元测试 | x.com/home → 扫读，x.com/xxx/status/xxx → 细读 |
| **推特实测-分块生效** | **浏览器验收** | **打开 X 首页 → 至少 80% 的英文长推文被拆分渲染** |
| **推特实测-即时响应** | **浏览器验收** | **本地拆分耗时 < 50ms/条推文（console 计时日志）** |
| 推特实测-滚动加载 | 浏览器验收 | 向下滚动 → 新加载的推文也被拆分（MutationObserver 工作正常） |
| 推特实测-LLM 降级 | 浏览器验收 | 复杂推文在 1-3 秒内显示 LLM 拆分结果 |
| 整体体感 | 人工确认 | 用户刷一屏推特，体感流畅，分块结果有助于理解 |

### 第 6 步：生词系统

| 验收项 | 类型 | 通过标准 |
|--------|------|----------|
| 词频过滤 | 单元测试 | 常用 5000 词不标注，超出词频表的词标注 |
| 行业术语匹配 | 单元测试 | AI 术语包中的词优先用行业释义 |
| 已知词跳过 | 单元测试 | 标记为"已掌握"的词不再标注 |
| 生词虚线显示 | 浏览器验收 | 被拆分的文本中，生词有 hover 虚线样式 |
| Hover 释义 | 浏览器验收 | 鼠标悬停虚线词 → 弹出释义浮层 |
| 释义准确性 | 人工确认 | 抽查 10 个生词，释义是否准确、是否符合语境 |

### 第 7 步：Popup

| 验收项 | 类型 | 通过标准 |
|--------|------|----------|
| Popup 渲染 | 浏览器验收 | 点击扩展图标 → Popup 正常显示，无空白或报错 |
| LLM 配置 | 浏览器验收 | 填入 API key、选择格式、保存 → 配置持久化（关闭再打开仍在） |
| 模式切换 | 浏览器验收 | 手动切换模式 → 页面行为跟随变化 |

### 第 8 步：IndexedDB 数据层

| 验收项 | 类型 | 通过标准 |
|--------|------|----------|
| CRUD 操作 | 单元测试 | 9 张表的增删改查均正常（使用 fake-indexeddb） |
| Schema 升级 | 单元测试 | 模拟版本升级，旧数据不丢失 |
| 数据持久化 | 浏览器验收 | 写入数据 → 关闭扩展 → 重新打开 → 数据仍在 |

### 第 9 步：Options 页面

| 验收项 | 类型 | 通过标准 |
|--------|------|----------|
| 页面渲染 | 浏览器验收 | 打开 Options 页 → React 应用正常加载 |
| 各模块可访问 | 浏览器验收 | Dashboard、生词本、阅读记录等页面均可切换，无白屏 |
| UI 和交互 | 人工确认 | 布局合理，操作符合直觉 |

## 测试数据

### 英文句子测试集

在 `tests/fixtures/sentences.json` 中维护一组固定的英文句子（从真实推特/Reddit 抓取），用于：
- scan-rules 单元测试的输入
- rule-engine 单元测试的输入
- 浏览器验收测试的对照参考

句子集应覆盖：
- 短句（< 10 词，不应被拆）
- 中等并列句（and/or/but 连接）
- 转折/条件/因果句
- 多层嵌套从句
- 信息密度高但语法简单的长句（不应被过度拆分）
- 中文推文（不应被处理）

### LLM 响应 mock

在 `tests/fixtures/llm-responses/` 中维护：
- `gemini-valid.json` — Gemini 格式正常响应
- `openai-valid.json` — OpenAI 格式正常响应
- `malformed.json` — 各种畸形响应（用于异常处理测试）

## 开发时的测试纪律

1. **编码前**：确认当前步骤的验收标准清单（查本文档对应章节）
2. **编码中**：核心逻辑模块先写单元测试，再写实现
3. **编码后**：
   - 跑 `npm test` 确保单元测试全绿
   - 跑对应的浏览器验收脚本确保真实效果
   - 标注哪些项需要人工确认，列出具体操作步骤给用户
4. **交付物**：代码 + 测试通过的日志/截图 + 人工确认清单
